# Headhunter AI

Десктопное приложение, автоматизирующее отклики на вакансии hh.ru с генерацией сопроводительных писем через AI.

> [!summary] **Текущий прогресс:** Stage 0 закрыт, Stage 1 (MVP) — большая часть собрана, осталась 1.10 (LetterReview UI). См. [[Roadmap]] и [[Stage 1 - MVP]].

## Суть

Снимает с соискателя рутину массовых откликов. Пользователь задаёт фильтр поиска и контекст о себе (резюме, желаемые скиллы, тон письма), приложение запускает Chromium, парсит подходящие вакансии в реальном времени и генерирует под каждую персонализированное сопроводительное письмо. Письма можно либо отправлять автоматически с лимитами, либо подтверждать вручную в собственном [[UI]].

## Ключевая архитектурная идея

В MVP backend генерирует письма через внутренний [[AI Layer]] (LiteLLM, multi-provider + fallback): пользователь приносит ключ от Anthropic / OpenAI или поднимает локальный Ollama. В настройках задаётся упорядоченная цепочка deployments — primary + fallbacks. Собственный [[UI]] взаимодействует с бэком через REST + WebSocket.

В [[Stage 2 - Расширение]] добавляется второй параллельный канал — [[MCP]]-сервер для внешних AI-клиентов (Claude Desktop, Claude Code), позволяющий обладателям Claude Pro/Max использовать подписку вместо per-token API. Оба канала пишут в одну БД.

## Целевая аудитория

- Активные соискатели, отправляющие десятки откликов в неделю
- Карьерные консультанты, помогающие клиентам

## Сценарии использования

- Массовый отклик по фильтру с автоотправкой
- Выборочные отклики с ручным ревью в [[UI]]
- Дозированная рассылка с лимитами (см. [[Anti-bot]])

## Исходное ТЗ

> Сохранено для истории. Конкретика разнесена по [[Functional Requirements]], [[Architecture]] и заметкам компонентов.

Клиент выполняет следующие шаги:
- При запуске [[Parser service]], который запускает [[Chromium]] — клиент должен авторизоваться на hh.ru
- Возможность настроить характер и контекст (знания о соискателе) чата
- Фильтр для поиска
- Клиенту отображается распарсенные вакансии:
    - Вакансия ждёт отправки сопроводительного письма
    - На вакансию был отправлен отклик и есть возможность посмотреть письмо
    - Любая другая ошибка

Бэк выполняет следующие шаги:
1. [[REST]] получает запрос с фильтром на поиск от клиента
2. [[Parser service]] запускает [[Chromium]] и определяет авторизован ли пользователь на hh.ru, иначе переводит его на форму авторизации
3. [[Parser service]] начинает парсинг каждой отдельной вакансии (поля см. в [[Domain Model]])
4. [[Parser service]] отдаёт вакансию в реальном времени (через WebSocket — FastAPI это поддерживает нативно, см. [[REST]])
5. [[REST]] принимает вакансию от [[Parser service]] и отдаёт её через MCP-tools для AI-клиента
6. AI-клиент формирует ответ для сопроводительного письма и дёргает MCP-tool `submit_cover_letter` (либо REST `/api/write/{vacancy_id}`)
7. [[Writer service]] откликается по ссылке с переданным письмом
